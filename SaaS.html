SaaS: One Size Fits All…


What is it?

Yes, you did read it correctly. Being a Software as a Service product, it must with a small number of UI components, serve a wide range of functionality for numerous teams. It's more like Legos, for lack of a better comparison. Although the UI components are straightforward, they can be combined to cater to a variety of user needs.


  • Intuitive, easy to understand action oriented object driven workflows
  • Low Code: Self building UI with robust in-house design system
  • Ability to self-serve for internal users and clients
  • Responsive enough for the users to configure on the go on mobile devices
  • Modern tool to sunset legacy applications


Why was there a need?

No one asked for it to change. But while I was supporting the current application I felt a need to give it a revamp. Some of the workflows were cumbersome to the users, some of the actions weren't obvious and moreover the accomplishments of the users were not celebrated.


  • Workflows that were highly error-prone and resulted in fines and reputational harm
  • Lack of consistency resulting in slowing the users tasks and productivity and increasing frustration
  • Difficult for the new development team to integrate due to namming confusion due ot lack of design thinking.


How it's done?

I therefore spent the time in three weeks going pixel by pixel to see what might be improved and learning what users wanted. What are the aches and pains? Which user flow is more logical? tried to find the unnecessary chores to automate. Make the computer work for you rather than the other way around.


  • Re-define each pixel to break new boundries
  • For every UI component, adhered to the BEM naming standard
  • 8-bit point system enforced to bring consistency to spacing
  • Up until the application moved into production, I closely collaborated with the dev team, stackholders, and POs.
